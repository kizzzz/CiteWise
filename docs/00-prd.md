# 产品需求文档（PRD）

**项目名称**：
**版本**：V1.0 MVP  
**目标**：为研究者提供从文献梳理到论文产出的一站式智能辅助，通过可定制结构化总结、强制溯源生成和个性化记忆，提升学术写作效率。

---

## 1. 项目背景

### 1.1 行业背景
学术研究和企业研发中，文献梳理与论文写作占据研究人员60%以上的时间。随着论文数量爆炸式增长，研究者面临“读不完、写不出”的困境。现有工具要么只做文献管理（Zotero），要么只做浅层问答（Elicit），要么只做通用生成（ChatGPT），缺乏一个打通“从阅读到写作”全流程的智能助手。

### 1.2 用户痛点

| 阶段 | 痛点 | 描述 |
|------|------|------|
| 文献管理 | 导入繁琐 | 手动整理PDF、重命名、分类，元数据不完整 |
| 文献梳理 | 结构化总结耗时长 | 每篇文献需人工提取关键字段（方法、数据、结论），不同学科关注字段不同，通用模板不适用 |
| 思路构建 | 框架设计缺乏依据 | 不知道按什么逻辑组织综述，难以发现多篇论文的共同点 |
| 论文写作 | 生成内容缺乏溯源 | ChatGPT等容易产生幻觉，捏造文献或数据；图表需手动截图、编号、引用，容易出错 |
| 迭代复用 | 重复配置 | 每个新项目都要重新设置字段偏好，知识无法跨项目复用 |

### 1.3 现有解决方案的不足

| 类型 | 代表产品 | 局限性 |
|------|----------|--------|
| 文献管理 | Zotero、Mendeley | 无内容理解，无生成能力 |
| 文献发现 | Connected Papers | 无内容理解，无写作辅助 |
| AI学术问答 | Elicit、Scispace | 字段固定，无完整论文生成，无跨会话记忆 |
| AI学术写作 | 掌桥科研AI、Kimi | 依赖自有数据库，无用户私有知识库记忆 |
| AI知识库 | NotebookLM | 偏探索而非产出，总结不可定制，无跨会话记忆 |
| 通用大模型 | ChatGPT、Claude | 无引用溯源，幻觉率高，无私有知识库 |

### 1.4 产品定位

**CiteWise** 是一个以用户私有知识库为核心的智能研究助手，帮助研究者从文献梳理到论文产出的全流程，通过可定制的结构化总结、强制溯源生成和个性化记忆，让学术写作效率提升5倍。

## 2. 用户画像

| 用户类型 | 特征 | 核心诉求 |
|----------|------|----------|
| 博士生/研究生 | 正在撰写学位论文，已收集10-30篇核心文献 | 快速梳理文献、构建综述框架、生成符合学术规范的章节 |
| 青年科研人员 | 需要持续产出论文/报告，有多个研究项目 | 跨项目复用文献库和生成模板，系统记住研究偏好 |
| 跨学科研究者 | 需要快速进入新领域 | 系统自动分析文献库并推荐研究框架，提取核心方法 |

## 3. 核心差异化

1. **从“读”到“写”的闭环**：文献梳理 → 思路讨论 → 分节生成 → 局部修改 → 完整论文产出。
2. **可定制的结构化总结**：用户自定义提取字段（如“研究方法”“核心发现”），批量生成表格和数据图。
3. **强制溯源防幻觉**：所有生成内容必须关联至少一篇知识库文献，无依据时明确告知。
4. **个性化记忆（跨会话）**：全局用户画像记录研究领域、字段偏好、关注点，跨项目复用。
5. **Agent内化写作工具链**：集成字段提取、表格生成、图表绘制、局部修改、引用格式化等能力。
6. **多模态图表处理**：输入时自动解析PDF中的图表并生成描述，输出时用户确认后自动嵌入文章。

## 4. 用户交互流程（六阶段）

### 4.1 第一阶段：项目创建与知识库构建
- 用户创建项目，上传PDF文献（10-30篇）
- 系统自动解析：按章节切片，提取文本和图表标题，调用多模态模型生成图表描述，存入向量库
- 输出：项目创建成功，提示可进行结构化总结

### 4.2 第二阶段：结构化总结、图表索引与可视化
- 用户自定义字段（如“方法名称”“骨干网络”“数据集”“mAP”“创新点”）
- Agent规划任务：对每篇文献检索对应字段，提取图表列表，生成结构化表格和数据图（饼图、柱状图）
- 输出：结构化表格（JSON/Markdown） + 图表索引 + 可视化图表

### 4.3 第三阶段：讨论研究思路，确定研究框架
- 系统主动分析表格数据，推荐顶刊标准框架（如：摘要、引言、文献综述、方法论、结果、讨论、结论）
- 用户可修改框架（如增加子主题）
- 系统记录用户偏好（如“重点突出空间异质性”）

### 4.4 第四阶段：Agent定制生成，完成完整文章
- 系统生成配置器：为每个章节配置检索策略、Prompt模板、输出格式、图片嵌入选项
- 用户确认配置后，Agent按顺序调用工具生成各章节，分节生成时通过滑动窗口摘要保持上下文连贯
- 输出：完整的综述文章（Markdown/Word），含正文、表格、图表、引用、参考文献

### 4.5 第五阶段：局部修改与图表调整
- 用户对某段文字或图表提出修改（如“把方法分类合并”“生成对比图”）
- Agent定位目标位置，调用修改工具重新生成，保持引用不变
- 输出：更新后的文章

### 4.6 第六阶段：跨项目复用与个性化推荐
- 用户新建项目，系统根据历史偏好主动建议：“是否沿用之前的字段模板？需要生成同样的图表吗？”
- 用户一键复用，快速开始新项目

### 5.2 核心能力实现

| 能力 | 技术实现 | 关键设计 |
|------|----------|----------|
| **RAG** | 层级切片（按章节），混合检索（BM25 + 向量），重排序（BGE-reranker） | 保留元数据（论文ID、章节、图表编号），支持图文混合检索 |
| **上下文工程** | 滑动窗口摘要，分节生成时传递前文压缩摘要 | 长文档生成时保持连贯，支持局部修改定位 |
| **Prompt Engineering** | 动态组装system prompt，按章节类型使用不同模板（引言、方法、结果等） | few-shot示例控制输出格式（JSON表格、Markdown） |
| **Agent** | ReAct模式，工具集（检索、表格、图表、修改、导出） | 多步规划，主动推荐，用户确认后执行 |
| **记忆工程** | 全局用户画像（研究领域、字段模板、关注点），会话状态（焦点论文、对话摘要） | 隐式/显式更新，跨项目复用 |
| **多模态（图片）** | 多模态模型（Qwen-VL）生成图表描述，存入向量库；输出时嵌入用户确认的图表 | 图表可检索，引用可溯源 |
| **长上下文** | 分节生成 + 滑动窗口摘要，每节独立调用但传递关键信息 | 支持15篇长论文（450+页）和5000+字综述 |

### 5.3 技术选型（MVP）

| 组件 | 选型 | 说明 |
|------|------|------|
| 前端 | Streamlit / Chainlit | 快速搭建对话界面 |
| 后端 | Python FastAPI | 轻量级API服务 |
| 向量库 | Chroma / Qdrant | 本地部署，支持混合检索 |
| Embedding | BGE-large-zh | 中文学术场景优化 |
| 重排序 | BGE-reranker | 提升检索精度 |

| 文件解析 | PyPDF2 + pdfplumber | 提取文本和图表 |

## 6. 功能需求

### 6.1 MVP核心功能

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 知识库构建 | 上传PDF，自动解析、切片、向量化 | P0 |
| 结构化总结 | 用户自定义字段，批量提取，生成表格 | P0 |
| 数据可视化 | 从表格生成饼图、柱状图 | P1 |
| 图表索引 | 提取论文中的图表并生成缩略图列表 | P1 |
| 思路讨论 | 系统主动分析数据并推荐论文框架 | P0 |
| 分节生成 | 按框架生成引言、文献综述、方法论等章节 | P0 |
| 强制溯源 | 每个观点附带引用来源 | P0 |
| 局部修改 | 基于对话修改段落或图表 | P1 |
| 记忆复用 | 跨项目保留字段模板和偏好 | P1 |
| 导出 | 导出Markdown/Word文档 | P1 |

### 6.2 非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | 单篇论文解析 ≤ 10s；单次问答 ≤ 5s；生成一节 ≤ 30s |
| 隐私 | 用户数据仅本地存储，可一键删除；使用云端API时数据不用于训练 |
| 成本 | MVP预估单用户月成本 ≤ $3（使用开源模型可降至 $1） |
| 可扩展 | 支持后续接入更多文件格式（Word、PPT）和外部数据库（arXiv API） |

## 7. 验收标准

| 测试项 | 标准 |
|--------|------|
| 结构化总结 | 对10篇测试论文，用户自定义5个字段，提取准确率 ≥ 85%（人工评估） |
| 强制溯源 | 生成内容中每个观点至少引用1篇知识库文献，引用准确率 ≥ 90% |
| 局部修改 | 用户提出“修改第3段”，系统能正确定位并重新生成，保留其他段落不变 |
| 记忆复用 | 新项目创建后，系统能主动提示使用历史偏好，用户确认后自动应用 |
| 多模态图表 | 上传含图表的PDF，系统能提取并生成描述；用户要求嵌入图表时，能正确插入并编号 |

## 9. 风险与应对

| 风险 | 应对策略 |
|------|----------|
| 多模态图表描述不准确 | MVP阶段仅生成简单描述（如图表标题+类型），复杂图表由用户确认 |
| 长文档生成上下文丢失 | 采用分节生成+滑动窗口摘要，每节独立调用但传递关键信息 |
| 幻觉率仍存在 | 强制引用机制 + 用户可手动验证引用来源 |
| 成本超支 | 提供本地模型选项（Qwen + BGE），用户可选择自部署 |
